HTML

  • Most meta elements are not visible to the user (except title)
  • h1 headers should only be used once per webpage
  • Almost all HTML elements have an opening and closing tag
  • Formatting the coding itself with line breaks, indents, etc do not affect the appearance of the final product
  • The head element contains information about the webpage
  • The body element represents the visible content shown to the user
  • Elements nested under an element (parent) are referred to as a child element
  • You can apply classifications to sections (class), such as cards. This makes it easy to apply CSS attributes to those sections in particular

CSS

  • CSS is the code that tells the browser what the content should look like
  • CSS work is referred to as front-end development (UI)
  • The HTML file needs to contain a link in the header to recognize its connection to the CSS file
  • Margin indicates how much space we want around the outside of an element
  • Padding indicates how much space we want around the content inside an element
  • You can assign a class to sections of your HTML body, and then apply CSS rules to those specific classes
  • CSS rules can be applied to classes, or to specific elements such as footer, h1, etc.
  • Colors can be applied to fonts, backgrounds, etc. through various codes such as rgb, hex, etc.

Git

  • A repo (repository) is where all of the files for a project go
  • Repos contain the coding files, the README file, and any additional assets to make the product work
  • Coding and modifications should always happen on a different branch from the main
  • To merge coding to the main branch, a pull request needs to be submitted, approved by the team, and then finally merged into the main branch
  • When approving a pull request, always double-check to make sure the correct branches are being merged in the right direction (working branch to the main branch)

(Bash Git Flow)

    Before sitting down to code:
  1. git status: checks what branch we are currently working on in bash
  2. git pull origin main: pull the remote main branch into the local main branch to sync up the local and remote versions
  3. git checkout |name of branch|: change git flow to an existing branch
  4. git checkout -b |branch-name|: creates a new branch and switches to it

    After finishing your coding:
  1. git status: double check to make sure you're still in the right branch!
  2. git add -A: stage the changes you made on your feature branch
  3. git commit -m "|a brief summary of the branch|": commit your changes to the repository, and provide a brief summary of what this new branch is adding to the project
  4. git pull origin main: make sure you local main branch and remote main branch are in sync
  5. git push origin |name of branch|: send the pull request to GitHub

JavaScript

  • JavaScript adds interactivity to webpages
  • For JavaScript coding, the order of your lines matters; for example, you need to first define your variable before logging it to the console
  • A value is considered falsy if it is one of the following values: 0, -0, On, "", null, undefined, NaN, or a Boolean data type with the value of false. All other values, including a Boolean with the value of true, are considered truthy. (lesson 3.3.4)
  • A variable is a named "container" that allows us to store data in our code (in the case of this study guide, the variable is TOPICS, and the TOPICS variable contains HTML, CSS, Git, and JavaScript
  • Control flow is the order in which a computer executes code in a script
  • You can randomize which choice in an array is selected using MATH functions
